# Document Shapes

Use these as loose starting points, not fixed templates. Rename sections, merge sections, or skip anything that does not help the current project.

## Master Plan

Describe the final imagined software: what it should do, who it serves, complete product requirements, chosen or recommended technology stack, why that stack fits, architecture direction, implementation path, MVP outcome, durable constraints, and major risks.

Make it complete enough for Product, Engineering, and Review to share the same destination. Keep detailed execution in the current iteration plan instead of expanding many future iteration plans.

## Current Iteration Plan

Describe the current iteration's goal, scope, related Spec files, acceptance criteria, engineering notes, and review focus.

Plan only the current iteration. Leave later iteration details until the current one is built, accepted, and reviewed.

## Spec

Describe the feature goal, scope, user-facing behavior, useful examples, acceptance criteria, edge cases, and current status.

Make the Spec clear enough for Engineering to implement without guessing.

## Review Output

Lead with a verdict: `approved`, `approved-with-comments`, `request-changes`, or `blocked`.

Then list the most important findings, risks, suggested changes, and evidence checked. Keep it practical and actionable.
